## ADDED Requirements

### Requirement: Spawn multiple cottons

The system SHALL support spawning up to 10 cottons simultaneously. Each cotton SHALL have a unique Gazebo model name (`cotton_0`, `cotton_1`, ...) generated by a monotonic counter that never resets during a session. Each cotton SHALL persist in Gazebo until explicitly removed via `POST /api/cotton/remove` or `POST /api/cotton/remove-all`. Pick sequences SHALL NOT delete cotton models.

#### Scenario: Spawn 3 cottons with different coordinates

- **WHEN** the user submits 3 different camera coordinate sets via `POST /api/cotton/spawn`
- **THEN** 3 distinct Gazebo models exist (`cotton_0`, `cotton_1`, `cotton_2`)
- **AND** `GET /api/cotton/list` returns all 3 with their coordinates and status `"spawned"`

#### Scenario: Counter does not reset after remove

- **WHEN** `cotton_0` is spawned, then removed, then a new cotton is spawned
- **THEN** the new cotton is named `cotton_1` (not `cotton_0`)

#### Scenario: Cotton persists after pick

- **WHEN** a cotton is spawned and then picked via `POST /api/cotton/pick`
- **THEN** the cotton's status changes to `"picked"`
- **AND** the Gazebo model still exists in the scene (not deleted)
- **AND** `GET /api/cotton/list` shows the cotton with status `"picked"`

#### Scenario: Only manual remove deletes from Gazebo

- **WHEN** a cotton with status `"picked"` exists
- **AND** the user calls `POST /api/cotton/remove` with that cotton's name
- **THEN** the Gazebo model is deleted
- **AND** the cotton is removed from the internal collection

### Requirement: Cotton list endpoint

The `GET /api/cotton/list` endpoint SHALL return a JSON array of all cottons in spawn order with fields: `name`, `cam_coords`, `arm_coords`, `joint_values`, `status`.

#### Scenario: Empty list when no cottons

- **WHEN** no cottons are spawned
- **THEN** `GET /api/cotton/list` returns `{"cottons": []}`

#### Scenario: List reflects current state

- **WHEN** 2 cottons are spawned and the first has been picked
- **THEN** the list returns cotton_0 with status `"picked"` and cotton_1 with status `"spawned"`

### Requirement: Sequential pick all cottons

The `POST /api/cotton/pick-all` endpoint SHALL pick all cottons with status `"spawned"` grouped by arm (see multi-arm-simultaneous-pick spec). For each cotton, it SHALL execute the full 5.5s pick animation. After the last cotton on each arm is picked, that arm SHALL return to home position (J3=0, J4=0, J5=0). Cotton models SHALL NOT be deleted from Gazebo after pick; only status is updated to `"picked"`.

#### Scenario: Pick 3 cottons sequentially

- **WHEN** 3 cottons are spawned and `POST /api/cotton/pick-all` is called
- **THEN** cotton_0 is picked first (5.5s), then cotton_1 (5.5s), then cotton_2 (5.5s)
- **AND** each cotton's status changes from `"spawned"` to `"picking"` to `"picked"`
- **AND** each arm returns home after its last cotton
- **AND** all 3 cotton Gazebo models still exist in the scene

#### Scenario: Skip already-picked cottons

- **WHEN** cotton_0 has status `"picked"` and cotton_1 has status `"spawned"`
- **AND** `POST /api/cotton/pick-all` is called
- **THEN** only cotton_1 is picked (cotton_0 is skipped)

#### Scenario: No cottons to pick

- **WHEN** all cottons have status `"picked"` or no cottons exist
- **AND** `POST /api/cotton/pick-all` is called
- **THEN** the endpoint returns `{"status": "nothing_to_pick"}`
- **AND** no arm motion occurs

### Requirement: Pick-all status endpoint

The `GET /api/cotton/pick-status` endpoint SHALL report progress during a multi-pick sequence with fields: `status`, `current` (cotton name being picked), `progress` (e.g., `"2/5"`).

#### Scenario: Status during multi-pick

- **WHEN** 3 cottons are being picked and the arm is on cotton_1
- **THEN** `GET /api/cotton/pick-status` returns `{"status": "picking", "current": "cotton_1", "progress": "2/3"}`

#### Scenario: Status after all picks complete

- **WHEN** all cottons have been picked and arm has returned home
- **THEN** `GET /api/cotton/pick-status` returns `{"status": "done", "current": null, "progress": "3/3"}`

### Requirement: Remove all cottons

The `POST /api/cotton/remove-all` endpoint SHALL remove all cottons from the internal collection AND delete all corresponding Gazebo models. The cotton counter SHALL NOT reset.

#### Scenario: Remove all after spawning 3

- **WHEN** 3 cottons exist and `POST /api/cotton/remove-all` is called
- **THEN** all 3 Gazebo models are deleted
- **AND** `GET /api/cotton/list` returns `{"cottons": []}`
- **AND** the next spawn creates `cotton_3` (counter continues)

#### Scenario: Remove all during pick sequence

- **WHEN** a pick-all sequence is in progress
- **AND** `POST /api/cotton/remove-all` is called
- **THEN** the endpoint returns `400 {"error": "Cannot remove during pick sequence"}`

### Requirement: Multi-cotton table UI

The Cotton Placement panel SHALL display a table showing all spawned cottons with columns: Name, Camera Coords, Joint Values (J3/J4/J5), Status. The table SHALL update after each spawn, pick, and remove operation.

#### Scenario: Table shows spawned cottons

- **WHEN** 2 cottons are spawned
- **THEN** the table shows 2 rows with status "Spawned" and their coordinates

#### Scenario: Table updates during pick

- **WHEN** a multi-pick sequence is running and cotton_0 completes
- **THEN** the table shows cotton_0 as "Picked" and cotton_1 as "Picking"

### Requirement: Remove All button

The Cotton Placement panel SHALL include a "Remove All" button that calls `POST /api/cotton/remove-all`. The button SHALL be disabled during a pick sequence.

#### Scenario: Remove All clears table

- **WHEN** the user clicks "Remove All" with 3 cottons in the table
- **THEN** all rows are removed from the table
- **AND** a status message confirms removal

#### Scenario: Remove All disabled during pick

- **WHEN** a pick-all sequence is in progress
- **THEN** the "Remove All" button is visually disabled and unclickable
